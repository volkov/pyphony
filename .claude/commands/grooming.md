# Grooming — интерактивный разбор бэклога

Интерактивный grooming тикетов из Linear. Агент предлагает действия, пользователь решает.

## Когда использовать

Trigger phrases:
- "груминг"
- "давай разберём бэклог"
- "grooming"
- "пройдёмся по задачам"

## Аргументы

$ARGUMENTS — опциональный фильтр: состояние тикетов (например "Backlog", "Todo") или конкретные идентификаторы (SER-XX). Если не указано — берём все тикеты.

## Алгоритм

### Шаг 1: Загрузка тикетов

Выполни:

```bash
./pyphony list-candidates
```

Это вернёт JSON со всеми тикетами проекта, их статусами и блокерами.

### Шаг 2: Обзор и фильтрация

Покажи пользователю краткую сводку загруженных тикетов в виде таблицы:

```
| # | Тикет   | Название                    | Статус   |
|---|---------|---------------------------- |----------|
| 1 | SER-10  | Добавить кэширование        | Backlog  |
| 2 | SER-15  | Исправить логирование       | Backlog  |
| 3 | SER-20  | Новый API endpoint          | Todo     |
```

Спроси пользователя:
- Какие тикеты разбираем? (все / только Backlog / конкретные номера)
- С какого начнём?

**Жди ответа пользователя. Не продолжай без него.**

### Шаг 3: Интерактивный обход тикетов

Для каждого выбранного тикета:

1. **Загрузи полные данные**:
   ```bash
   ./pyphony get-issue SER-XX
   ```

2. **Покажи тикет** — название, описание, статус.

3. **Предложи действия** (одно или несколько):
   - **Декомпозиция**: если задача слишком большая — предложи разбить на подзадачи, перечисли конкретные подзадачи
   - **Уточнение описания**: если описание пустое или нечёткое — предложи улучшенный вариант
   - **Смена статуса**: если задача готова к работе — предложи перевести в Todo
   - **Добавление лейблов**: `plan required` (нужен план), `research` (нужно исследование), `interactive` (ручная работа)
   - **Пропуск**: тикет в порядке, идём дальше

4. **Жди решения пользователя.** Варианты:
   - "да" / "давай" / "засыпай" — выполни предложенные действия
   - "нет" / "пропусти" / "skip" — переходи к следующему тикету
   - Пользователь может предложить свой вариант — следуй его указаниям
   - "стоп" / "хватит" — заверши grooming

### Шаг 4: Применение действий

Выполняй действия **только после явного подтверждения** пользователя.

**Создание подзадач:**
```bash
./pyphony create-issue --title "Название подзадачи" --description "Описание" --state "Backlog"
```

**Обновление тикета (описание, статус):**
```bash
./pyphony update-issue SER-XX --description "Новое описание"
./pyphony update-issue SER-XX --state "Todo"
```

**Добавление лейблов:**
```python
uv run python -c "
import asyncio
from pyphony.workflow import load_workflow
from pyphony.config import service_config_from_workflow
from pyphony.tracker import LinearClient

async def main():
    wf = load_workflow('WORKFLOW.md')
    cfg = service_config_from_workflow(wf.config)
    client = LinearClient(cfg)
    await client.replace_issue_labels('<issue_id>', remove_labels=[], add_labels=['<label_name>'])
    await client.close()

asyncio.run(main())
"
```

Подставь реальные `issue_id` (поле `id` из get-issue, НЕ identifier) и `label_name`.

После каждого действия покажи результат и переходи к следующему тикету.

### Шаг 5: Итог

После завершения обхода покажи сводку:
- Сколько тикетов разобрали
- Что создали (новые подзадачи)
- Что обновили (описания, статусы, лейблы)
- Что осталось нетронутым

## Важные правила

1. **Никаких действий без явного подтверждения пользователя** — ты предлагаешь, пользователь решает
2. **Язык общения — русский**
3. **Не торопись** — давай пользователю время на обдумывание каждого тикета
4. **Можно прервать в любой момент** — grooming не обязательно проходить до конца
5. **Будь конкретным** — не "можно разбить на подзадачи", а "предлагаю разбить на: 1) ..., 2) ..., 3) ..."
6. **Используй контекст** — если видишь связанные тикеты, упоминай их
